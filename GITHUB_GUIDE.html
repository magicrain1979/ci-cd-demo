GitHub 核心操作完全指南

以 ci-cd-demo 项目为实例，零基础到能干活

Issue 提需求 → Branch 开分支 → Commit 提交 → Pull Request → Actions CI → Merge 合并

一、仓库 (Repository)

仓库就是项目的"家"。所有代码、文档、Issue、PR 都住在这里。你的演示仓库地址：https://github.com/magicrain1979/ci-cd-demo

repo 克隆与本地操作

1.1 克隆仓库到本地

# HTTPS 方式（推荐新手）
git clone https://github.com/magicrain1979/ci-cd-demo.git

# 进入项目
cd ci-cd-demo

1.2 日常提交三部曲

# 1. 看改了哪些文件
git status

# 2. 暂存改动
git add .                     # 添加所有改动
git add src/index.js          # 或只添加某个文件

# 3. 提交（写清楚干了什么）
git commit -m "feat: 添加除法运算的除零保护"

# 4. 推送到 GitHub
git push
提交信息规范：feat: 新功能 / fix: 修 bug / docs: 文档 / test: 测试 / chore: 杂项

1.3 分支操作（核心技能）

# 创建并切换到新分支
git checkout -b feature/add-multiply

# 查看所有分支
git branch -a

# 切回主分支
git checkout master

# 推送新分支到 GitHub
git push -u origin feature/add-multiply

# 删除已合并的本地分支
git branch -d feature/add-multiply

1.4 同步与拉取

# 拉取远程最新代码
git pull

# 拉取所有远程分支信息
git fetch

# 强制覆盖本地（慎用！）
git reset --hard origin/master

二、Issue（问题 / 需求跟踪）

Issue 是 GitHub 的"任务系统"。提 bug、提需求、讨论方案都用它。

issue 从创建到关闭全流程

2.1 创建 Issue（网页操作）

1打开 https://github.com/magicrain1979/ci-cd-demo/issues
2点击绿色 New issue 按钮
3标题示例：加法计算器结果不准确
4正文用 Markdown 描述，可贴代码、截图

2.2 Issue 正文模板（Markdown）

## 描述
add(0.1, 0.2) 返回 0.30000000000000004，期望 0.3

## 复现步骤
1. 运行 `node src/index.js`
2. 调用 `add(0.1, 0.2)`

## 期望行为
返回 0.3

## 环境
- Node.js v20
- Windows 11

2.3 用 gh 命令操作 Issue

# 创建 issue
gh issue create --title "修复浮点数精度问题" --body "add(0.1,0.2) 精度不准"

# 列出所有 issue
gh issue list

# 查看某个 issue（#1 是编号）
gh issue view 1

# 关闭 issue
gh issue close 1
Issue 怎么关？ 在 commit 信息里写 fixes #1 或 closes #1，代码合并后 Issue 自动关闭。也可以在网页上手动点 Close issue。

三、Pull Request（拉取请求 / 代码审查）

PR 是 GitHub 的灵魂。你改了代码 → 发起 PR → 别人审查 → 测试通过 → 合并到主分支。

pull request 实战：从分支到合并

3.1 完整流程（命令 + 网页）

# ① 创建功能分支
git checkout -b fix/floating-point

# ② 修改代码... 然后提交
git add src/index.js
git commit -m "fix: 修复浮点数加法精度问题，fixes #1"
git push -u origin fix/floating-point

# ③ 去网页创建 PR：
#    https://github.com/magicrain1979/ci-cd-demo/pulls
#    点击 New pull request
#    base: master  ← compare: fix/floating-point
#    填写标题和描述，点击 Create pull request

3.2 PR 描述模板

## 改动内容
使用 Number.EPSILON 处理浮点数比较，解决精度问题

## 测试
- add(0.1, 0.2) → 0.3 ✓
- 原有测试全部通过

## 关联 Issue
Closes #1

3.3 审查与合并

# 审查者可以在线添加评论，点 + 号选行评论

# 所有检查通过后，合并方式有三种：
# ① Merge commit    — 保留完整历史（推荐团队项目）
# ② Squash & merge  — 多条 commit 压成一条（推荐个人项目）
# ③ Rebase & merge  — 线性历史，不产生 merge commit

# 合并后，删除远程分支
git push origin --delete fix/floating-point

# 本地切回主分支并同步
git checkout master
git pull

3.4 用 gh 命令操作 PR

# 创建 PR（push 分支后）
gh pr create --title "修复浮点数精度" --body "Closes #1"

# 列出 PR
gh pr list

# 查看 PR 详情和 CI 状态
gh pr view

# 合并 PR
gh pr merge --squash

# 查看某个 PR 的 checks（CI 状态）
gh pr checks

四、Actions（CI/CD 自动化）

代码 push 或提 PR 时，自动跑测试、构建、部署。配置写在 .github/workflows/ 目录下。

actions 你的 ci-cd-demo 项目就是他

4.1 查看流水线运行

# 在网页上
https://github.com/magicrain1979/ci-cd-demo/actions
# 点进每次运行，可以看到每一步的实时日志

# 用命令查看
gh run list                          # 列出最近的运行
gh run view                          # 交互式选择查看某次运行
gh run watch                         # 实时看日志滚动

4.2 手动触发

gh workflow run "CI/CD Pipeline"   # 手动触发一次

4.3 CI 失败时怎么排查

1打开 Actions页面，点红色的运行记录
2展开失败的 job，查看哪一步红了
3点开那一步，看终端输出日志
4根据日志提示修代码，commit + push，自动重跑

4.4 你的 ci.yml 解读

name: CI/CD Pipeline          # 流水线名称

on:                           # 什么时候触发
  push:
    branches: [main]          # push 到 main 时
  pull_request:
    branches: [main]          # 提 PR 到 main 时

jobs:
  ci:                         # 一个叫 ci 的任务
    runs-on: ubuntu-latest    # 在 Ubuntu 虚拟机上跑
    steps:                    # 按顺序执行
      - uses: actions/checkout@v4      # 拉代码
      - uses: actions/setup-node@v4    # 装 Node.js
      - run: npm install               # 装依赖
      - run: npm run lint              # 代码检查
      - run: npm run build             # 构建
      - run: npm test                  # 跑测试

# 创建 Release（网页或命令）
gh release create v1.0.0 --title "v1.0.0 首次发布" --notes "新增加减乘除功能"

settings 仓库设置要点

设置项位置说明
默认分支Settings → General → Default branch通常设为 main 或 master
分支保护Settings → Branches → Add rule要求 PR 必须审查、CI 必须通过
Actions 权限Settings → Actions → General控制谁能触发 CI
CollaboratorsSettings → Collaborators添加团队成员
